## Part 1 — Critical Security Fix (Do This First)

### The bug

A regular user selects the Admin tab on the login page and successfully enters the admin dashboard. The backend is only checking the password — it is not independently verifying whether the user's actual role in the database matches what they claimed on the frontend.

### Broken flow (current)

```
User account → role: USER in database
    ↓ selects Admin tab on login page
    ↓ enters correct credentials
    ↓ backend checks password only — correct
    ↓ LOGIN SUCCEEDS — full admin access granted
    ↓ user is inside admin dashboard illegitimately
```

### Correct flow (must implement)

```
User account → role: USER in database
    ↓ selects Admin tab — sends login_as: "admin" in request body
    ↓ enters correct credentials
    ↓ backend checks password — correct
    ↓ backend fetches role from DB — role is USER
    ↓ backend compares DB role vs login_as — MISMATCH
    ↓ returns 403 ROLE_MISMATCH
    ↓ frontend shows error — login blocked completely
```

### Fix 1A — Update POST /auth/login to validate in this exact sequence

```
Step 1  Validate credentials (username + password)
        → wrong credentials → return 401

Step 2  Fetch user record from database

Step 3  Read role from database (USER / ADMIN / MASTER)

Step 4  Compare DB role against login_as from request body
        → mismatch → return 403 ROLE_MISMATCH

Step 5  If role is ADMIN → check account status column in DB
        → status is PENDING  → return 403 ACCOUNT_PENDING_APPROVAL
        → status is REJECTED → return 403 ACCOUNT_REJECTED
        → status is ACTIVE   → continue to step 6

Step 6  Issue JWT token with role embedded from database (not from frontend)
```

### Fix 1B — Request body must include login_as field

Frontend must send this on every login attempt:

```json
{
  "username": "neo10486",
  "password": "••••••••",
  "login_as": "user"
}
```

The active tab (Employee = `"user"`, Admin = `"admin"`) sets the `login_as` value. Backend compares this against the real role in the DB and rejects if they do not match exactly.

### Fix 1C — JWT token must embed real role from database

```json
{
  "user_id": "abc123",
  "username": "neo10486",
  "role": "USER",
  "status": "ACTIVE",
  "exp": 1234567890
}
```

### Fix 1D — All /admin/* routes must verify JWT role server-side

```python
decoded = verify_jwt(token)
if decoded["role"] not in ["ADMIN", "MASTER"]:
    return 403, "ACCESS_DENIED — Insufficient privileges"
```

This check must happen on every single admin endpoint. Frontend cannot be trusted.

### Fix 1E — Frontend error messages

Show these in the existing red error box on the login page:

| Backend returns | Show on frontend |
|---|---|
| 401 | `AUTHENTICATION_FAILED — Invalid credentials` |
| 403 ROLE_MISMATCH | `ACCESS_DENIED — Account does not have admin privileges` |
| 403 ACCOUNT_PENDING_APPROVAL | `ACCESS_PENDING — Awaiting master account approval` |
| 403 ACCOUNT_REJECTED | `ACCESS_REJECTED — Account access has been denied` |

### New admin account — seed directly in database

Do not create this via the UI. Insert directly into the users table:

```sql
INSERT INTO users (username, email, password_hash, role, status, created_at)
VALUES (
  'skynet_admin',
  'admin@skynet.ai',
  '<bcrypt hash of SkyAdmin@2026>',
  'ADMIN',
  'ACTIVE',
  NOW()
);
```

```
Username : skynet_admin
Email    : admin@skynet.ai
Password : SkyAdmin@2026
Role     : ADMIN
Status   : ACTIVE
```

Also fix the old account that was used to bypass — correct its role to `USER` in the database immediately.

## Part 5 — Database Changes Required

### Users table — ensure these columns exist

```sql
ALTER TABLE users ADD COLUMN IF NOT EXISTS role VARCHAR(20) NOT NULL DEFAULT 'USER';
ALTER TABLE users ADD COLUMN IF NOT EXISTS status VARCHAR(20) NOT NULL DEFAULT 'ACTIVE';
ALTER TABLE users ADD COLUMN IF NOT EXISTS created_at TIMESTAMP DEFAULT NOW();
ALTER TABLE users ADD COLUMN IF NOT EXISTS ppt_count INTEGER DEFAULT 0;
```

### Generations table — store username on every generation

```sql
ALTER TABLE generations ADD COLUMN IF NOT EXISTS user_id VARCHAR(100);
ALTER TABLE generations ADD COLUMN IF NOT EXISTS username VARCHAR(100);
```

Every call to `POST /generate` must extract `user_id` and `username` from the JWT token and save them into the generation record. Without this, the `Generated by` label cannot work.

```python
# In the /generate route
user_id  = decoded_token["user_id"]
username = decoded_token["username"]
# Save both to generations table alongside the deck data
```

## Part 6 — All Required API Endpoints

All routes require `Authorization: Bearer <JWT>` header. All `/admin/*` routes must reject any JWT where `role` is not `ADMIN` or `MASTER`.

```
GET    /admin/stats                 → stat card numbers
GET    /admin/users                 → all registered users
GET    /admin/users/:id/ppts        → all PPTs for a specific user (for VIEW_PPTS action)
GET    /admin/generations           → all PPTs across all users
GET    /admin/pending               → list of pending admin approval requests
POST   /admin/users/create          → admin creates a new account
POST   /admin/approve/:userId       → master approves a pending admin account
POST   /admin/reject/:userId        → master rejects a pending admin account
PATCH  /admin/users/:id             → update user status (suspend, reactivate)
DELETE /admin/users/:id             → delete a user account
DELETE /admin/generations/:id       → delete a generation record
```
